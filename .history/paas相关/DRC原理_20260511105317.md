# DRC 原理

## 模块概述

DRC 主要由以下 7 个模块组成：

1. 中心节点与单元节点
2. DRC 管理端
3. 通用模块
4. 生产者模块 (Producer)
5. 消费者模块 (Consumer)
6. DRC 链路检测工具 (MySQL)
7. 数据校验模块

## 一、中心节点与单元节点

### 基本概念

- **中心节点**：通常指北京数据节点（MySQL、MQ、JIMDB 等）集群
- **单元节点**：通常指区域数据节点，如广州数据库（一主多从集群）

### 数据分布与同步原则

- 中心节点有全量数据，单元节点有本单元的数据
- 单元节点可读写本单元的数据
- 单元节点如有写入数据，需同步到中心节点
- 中心节点数据可同步到对应单元集群

## 二、DRC 管理端（前端 + 后端）

### 任务管理流程

用户通过前端创建同步任务，输入任务相关信息（源域名、目标端域名、过滤规则、同步方式等），后端负责：

- 将任务元数据信息存入 etcd 集群
- 将相关任务信息存入 MySQL 集群
- 在 etcd 中创建任务节点 key（一个 key 对应一个任务，不同 key 相互独立）
- 存储任务详情和任务状态

任务调度模块 (scheduler) 监听 etcd 中任务节点 key 的变化来调度任务执行。

### 任务可见性控制

前端在用户创建任务时，会将用户信息与该任务进行绑定，用户只能查看自己创建的任务，其他任务不可见。

### 基于图数据库的任务调度方法

**适用于：MySQL/JED 任务**

任务调度系统启动或重启时，查询出图数据库中所有符合执行条件的任务节点：

**执行条件：**
- 该任务状态不为完成
- 节点的计数器值为 0
- 该任务没有前置父任务，或所有前置父任务均已执行完成

单个任务开始执行后均遵循相应流程。



**任务执行过程说明：**

- 对任务状态的判断更新、对任务属性值的更新均为即时查询、更新操作
- 对某任务的前置父任务和后续子任务的查询也是即时进行
- 无需在系统内存中单独维护一整套任务流程图
- 只要每个任务的执行都遵循相应流程，即可保证整条任务链路的正常流转
- 单个任务的前置父任务数量没有限制

### 告警功能

- **邮件告警**：通过 etcd 元数据大 value 通知
- **咚咚告警**：通过 etcd 元数据大 value 通知

## 三、通用模块

### 核心功能

通用模块主要作用是通过读取 etcd 中的任务元数据，来控制：

- 任务的启动与暂停
- 任务数据的动态修改
- 任务状态的获取与更新

### 健康度调度

根据各个任务执行节点的健康度来进行任务调度。例如：

- 某个 producer 创建的任务很多，健康度降低
- 则在创建任务时选择创建任务比较少的执行节点

## 四、Producer 插件

### MySQL Producer

#### 前置要求

- 表必须有主键或唯一索引
- binlog 格式必须为 row 格式
- 必须设置配置参数：`binlog_rows_query_log_events=1`（否则无法进行单元数据过滤）

#### 核心功能

Producer 主要任务是从源端数据库拉取全量和增量数据，并通过 gRPC 协议发送给 consumer 插件。

#### 任务模式

- 全量数据同步
- 增量数据同步
- 全量 + 增量数据同步

#### 任务流程

**任务注册 → 任务订阅**

**任务注册阶段：**
- Producer 验证源端数据库的有效性
- 验证项：用户名密码、网络连通性、数据库连通性等
- 如果是增量任务，使用 binlog 协议建立连接，准备拉取 binlog

**任务订阅阶段：**
- 根据任务参数启动数据同步任务

#### 数据发送方式

- **协议**：gRPC
- **数据格式**：统一 protobuf 格式
- **优势**：
  - 流量压缩
  - 格式统一
  - 加快同步效率

#### 部署方式

Producer 插件以基础镜像的方式，直接部署在容器中。

#### 容器健康度检查

根据以下指标进行健康检查：
- 容器内存使用情况
- 容器磁盘使用情况
- CPU 使用情况
- 创建的任务数量

健康数据值存入 etcd，通用模块使用这个数据来进行节点间的调度。

#### 数据初始化方式

**全量任务模式 / 全量 + 增量任务模式：**

- 按照库表的顺序，每次 select 批量数据发送给 consumer
- 获取 select 执行期间产生的增量数据发送给 consumer
- 交替执行，直到全量数据初始化成功

**优势：**
- 初始化全量数据时降低源端数据库压力
- 防止增量数据量过大导致 binlog 数据被清理
- 加快数据同步流程
#### 流量统计与性能监控

**流量统计：**
- 统计从源端数据库获取的总流量
- 统计获取流量速率
- 接入 UMP 监控

**性能监控：**
- 程序中重要方法的可用率
- 调用量
- TP99 等指标
- 接入 UMP 监控
#### 过滤方式

- **维度过滤**：库、表、字段值过滤
- **单元过滤**：单元 (user pin) 过滤，以注释的方式拼在 SQL 语句后面

## 五、Consumer 插件

### MySQL Consumer

#### 核心功能

Consumer 的作用是：
- 接收从 producer 发送过来的数据
- 组装成 MySQL 协议可执行的 SQL 文件
- 将数据灌入目标数据库

#### 并发执行策略

Consumer 模式有两种并发策略。

### 并发执行策略 1：按表维度

以数据库的表为维度进行数据并发执行。

**具体执行步骤：**

1. Producer 在处理全量数据时，将每行数据处理成 MySQL insert 语句形式
2. Consumer 将每次从 producer 获取的一个批次数据以表为单位进行分别处理
3. 根据表的索引情况进行处理：
   - 表既有唯一索引也有主键：按策略 1 执行
   - 表只有主键或只有一个唯一索引：进行合并操作处理
4. 将合并后的操作在目标数据库执行

**合并策略：**

对于操作同一个主键或唯一索引的行数据：
- insert + update(1个或多个) = insert
- insert + delete = delete
- update(1个或多个) + delete = delete
- update + update(1个或多个) = update

**适用场景：**

满足数据最终一致性，当对数据实时性要求不是很高时可采用此策略。

### 并发执行策略 2：按事务维度

以事务为单位进行并发执行。

binlog 文件会记录数据库每条事务信息，根据事务信息，将 binlog 数据以事务为单位进行操作处理。

**具体步骤如下：**

1. Producer 在获取每批次全量数据后，将该批全量数据封装成事务发送给 consumer
2. Producer 将 binlog 增量数据解析处理，同样以事务维度发送给 consumer

#### Consumer 事务处理模块

**① 事务数据预处理**
- 将生产者发送的数据进行初步处理
- 解析每个事务中涉及的每行数据
- 获取每行数据的主键和唯一索引值

**② 事务分发器**
- 将处理过的事务数据定时或定量发送到事务冲突检测模块
- 避免事务一个一个发送导致串行执行
- 大事务单独执行（判断标准：事务中操作数据的行数）

**③ 事务冲突检测模块**
- 根据表的主键值和唯一索引值进行检测
- 对事务分发器发送的每批事务进行冲突检测
- 有冲突的事务不能并行执行，无冲突的事务可以并行执行

**④ 构造有向无环图**
- 将每一个事务作为图的一个节点
- 先构造无向无环图，再根据冲突检测结果添加有向边
- 有冲突的节点之间添加有向边（时间戳小指向时间戳大）
- 将无向无环图分成多个有向无环图
- 每个有向无环图内的相互不冲突事务可以并发执行
- 并发执行只能是相邻的节点
- 生成有向无环图的拓扑排序
- 按照拓扑排序和事务执行规则并发执行事务节点

## 六、DRC 链路检测工具 (MySQL)

### 功能

检测 producer 从源端数据库获取数据后到目标库获取该数据的整体延时情况。

### 实现方法

1. Consumer 记录获取到 producer 发送每条记录的时间
2. Consumer 接收到一条数据后，在目标库执行 `select 1` 操作，记录操作返回的时间
3. 两个时间相减得到数据库数据同步链路的大致延迟时间

## 七、数据校验模块 (MySQL)

### 功能

在全量数据同步的过程中，校验源端与目标端的数据是否一致。

### 实现方法

1. 解析目标端的 binlog 数据
2. 连接源端数据库，根据解析的 binlog 数据，在源端库中查找相应数据
3. 如未查到数据，则：
   - 将异常数据记录写入 ES
   - 在前端展示异常数据
   - 在初始化数据同步完成后进行处理

## 八、高可用

### MySQL 高可用

**Producer / Consumer 高可用：**

- 如果节点挂掉，通用端会立刻拉起一个新的任务节点
- 从之前节点记录的位点重新开始任务

## 十、为什么使用 Producer 与 Consumer 两个执行节点？

### 架构设计理由

**1. 开发解耦，架构独立**
- 插件模型最大的意义在于解耦和复用
- 只需提供一套基础框架，开发一系列同步插件
- 通过配置组合便可支持"无限多"的同步场景

**2. 网络优化**
- Producer 和 consumer 之间解耦
- 可以配置专线网络，加快数据传输

**3. 传输优化**
- 使用 gRPC 协议传输
- 压缩传输的数据
- 降低网络 IO
- 加快传输效率

**4. 功能扩展**
- 便于后续中间件数据异构功能的实现
